Data structure is complex.

Data is repeated.

  • Duplicated data is difficult to update.
  • Nested data increases complexity of the reducer implementation…
  • …and may cause unnecessary rendering.

Normalizing State

Each type of data gets its own "table"


{
  posts: { ... },
  comments: { ... },
  users: { ... }
}

Each "table" is an object. For each item, the item ID is the key and the value is item. 


{
  posts: {
    byId: {
      post1: { ... },
      post2: { ... }
    }
  },
  ...
}

Item order is stored in an array containing the item IDs.


{
  posts: {
    byId: {
      post1: { ... },
      post2: { ... }
    },
    ids: ['post1', 'post2']
  },
  ...
}

Related items are referenced by ID.


{
  posts: {
    byId: {
      post1: { author: 'user1' }
    }
  },
  users: {
    byId: {
      user1: { name: 'User One' }
    }
  }
}

Normalizing nested data

Data returned from APIs is often nested.

Normalizr is a small utility that simplifies converting nested JSON to a flat structure.
https://github.com/paularmstrong/normalizr

Authentication

Implemented like any other feature in your application.

State will typically include:

  • A flag to indicate whether the user is authenticated
  • The authentication token
  • The ID of the authenticated user
  • An error, if authentication failed

Use async action to handle sign in, e.g., LOGIN, LOGIN_SUCCESS, LOGIN_FAILURE. Sign out is typically a synchronous action for JWT, since it usually just involves removing the access token from client storage.
